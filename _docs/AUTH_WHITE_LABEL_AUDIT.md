## 0. Architecture, as it actually is

Understanding this first is what makes the rest of the document actionable.

| Layer | Vendor | Role in auth |
|---|---|---|
| Identity | Firebase Auth | Phone OTP, email/password, Google. **Owns all auth email + SMS.** |
| Data | Supabase Postgres | `users` table, RLS keyed on an exchanged JWT |
| Bridge | Supabase Edge Function `exchange-firebase-token` | Firebase ID token → Supabase JWT, so RLS sees `user_id` |
| API | NestJS on Render | M-Pesa, push, routes. **Not in the auth path.** |
| Client | Flutter | `AuthService` → `AuthProvider` → `AuthScreen` |

**The single most important consequence:** auth emails and the SMS body are generated by
Firebase, not by Help24 code. No amount of Dart can rebrand them. Sections 8–11 are
therefore console/DNS work, and they are where the remaining brand leakage lives.

## PHASE 2–3 — Stack leaks removed, errors made human

New file: **`lib/utils/auth_error_mapper.dart`**

Every provider error now passes through `AuthErrorMapper.toFailure()`, which returns an
`AuthFailure { title, message, recovery }`. **`e.message` is never returned** — it is
`debugPrint`-logged only. A deny-list backstop (`isSafeToShow`) blocks ~40 vendor and
protocol terms so a future SDK inventing new prose still cannot leak.

The two headline replacements:

> ~~"This request is missing a valid app identifier, meaning that Play Integrity checks and reCAPTCHA checks were unsuccessful."~~
> **"We couldn't verify your device"** — "We couldn't verify your device right now. Check your internet connection and try again. If this keeps happening, update Help24 from the Play Store." *(+ a "Get help" action)*

> ~~"Unable to process request due to missing initial state. This may happen if browser sessionStorage is inaccessible…"~~
> **"We couldn't finish signing you in"** — "We couldn't complete sign in. Please return to Help24 and try again."

27 provider codes are mapped explicitly. `test/auth_error_mapper_test.dart` feeds hostile
prose (`"FIREBASE INTERNAL: identitytoolkit rejected the OAuth token…"`) into **every**
code and asserts the output is clean — 66 tests.

## PHASE 4 — Sign In / Create Account: why users were confused, and the fix

**Diagnosis.** The screen opened on a `Sign In | Sign Up` segmented control *and* repeated
the same choice as a link below it. That asks: *"do you already have a Help24 account?"* —
a question a user genuinely cannot answer six months later, on a shared phone, having also
once tapped the Google button. Guessing wrong dead-ended in **both** directions.

**Fix — identity-first routing.** The toggle is deleted. The user types their email **once**:

```
Email  →  [Continue]  →  account exists? ──yes──→  Password        ("Welcome back")
                                         └──no───→  Create account  (name + password)
```

**No flow can dead-end**, because every `AuthFailure` carries a `recovery` action that the
`_FailureCard` renders as a **real button** which **carries the email forward**:

- Sign-in reports no account → *"No account yet"* + **[Create account]**
- Sign-up reports existing → *"You already have an account"* + **[Sign in instead]**
- Bad password → **[Reset password]** (opens the sheet pre-filled)

An `_EmailChip` shows the working email with a **Change** link, so a typo costs one tap
instead of restarting.

### ⚠️ One honest limitation

`fetchSignInMethodsForEmail` is **deprecated and neutered by email-enumeration
protection**, which is ON by default and deliberately returns "no methods" for every
address — it is an anti-harvesting defence and *should* stay on. So the pre-check returns
`AccountStatus.unknown` in most projects, and the flow routes optimistically to the
password step, then recovers via the buttons above.

**The user-visible promise holds either way** — the polite "we couldn't find an account"
+ **Create Account** button still appears, just after the attempt rather than before it.
Getting it *before* would require a server-side email-existence endpoint, which is
itself an enumeration oracle. Recommendation: **keep the current behaviour.**

## PHASE 5 — Email authentication, made production-grade

| Area | Before | Now |
|---|---|---|
| Email verification | **Never sent** | Sent on signup; `EmailVerificationBanner` nudges + resends; `refreshEmailVerified()` polls |
| Password reset | Leaked `user-not-found` | Returns success either way (**enumeration fixed**); copy is true either way; expiry + spam-folder guidance |
| Password rules | ≥6 chars | ≥8, rejects all-numeric, live 3-segment strength meter |
| Email validation | Rejected `.africa`, `.online` | RFC-shaped, any TLD ≥2 |
| Link destination | Provider default | `ActionCodeSettings` → `help24.co.ke/auth/continue` |
| Autofill | None | `AutofillHints` on email/password/newPassword/name |

Verification is a **nudge, not a wall** — blocking entry would tank activation for
phone-first Kenyan users. Escalate to hard enforcement at money-movement, not app entry.

## PHASE 6 — Kenyan phone input

New: **`lib/utils/kenyan_phone.dart`** + **`lib/widgets/auth/phone_number_field.dart`**

**Root cause of the `254`/`07`/`7` confusion:** `+254` was rendered as `prefixText`
*decoration* while the field still accepted every other shape. Users read it as a **label**
and typed their full local number underneath → `+2540712345678`, which the
`length < 12` check happily accepted.

**Fix — the country code is now structural, not decorative:**

- `+254` and 🇰🇪 live in a **non-editable gutter**, hairline-divided from the input
- The field holds **only the 9 national digits**; a `TextInputFormatter` collapses
  `0712…`, `254712…`, `+254 712…`, and the double-prefix `2540712…` to the same digits
- Letters/symbols are **impossible to enter**; a 10th digit is impossible
- Live grouping `712 345 678`; caret survives mid-number edits
- Prefix validation: Kenyan mobile = **7XX or 1XX** only (landlines rejected)
- Errors timed properly: **bad prefix → immediate** (further typing is wasted);
  **too short → only at submit** (nobody wants "too short" while typing)
- Green border + tick the instant the number is dialable; **Send code disabled until then**

51 tests in `test/kenyan_phone_test.dart` cover every input shape including the
double-country-code regression.

## PHASE 7 — OTP experience

New: **`lib/widgets/auth/otp_input.dart`**

Built as **one hidden field painted as six boxes** — deliberately *not* six `TextField`s,
which is the implementation that always ships with broken backspace, paste-fills-one-box,
autofill-drops-five-digits and six screen-reader announcements.

Delivered: paste ✓ · **SMS autofill** (`AutofillHints.oneTimeCode`) ✓ · **auto-submit on
6th digit** ✓ · caret in the next box ✓ · error shake + haptic + auto-clear ✓ ·
one labelled a11y field ✓ · **exponential resend backoff 30s → 60s → 120s** (a flat 30s
lets a user with no signal burn the SMS quota into a provider-level block) ✓ ·
"Use a different number" escape ✓ · Android instant-verification pass-through ✓.

## PHASE 8 — Why Help24 emails go to spam

**Confirmed: Firebase's default email service is sending everything.** Supabase sends no
auth mail (it is only the data layer). Every reason, in order of impact:

1. **Shared-IP sender.** Firebase's default pool is shared across countless projects; you
   inherit their reputation. **Biggest single factor.**
2. **From-address is `noreply@help24-24410.firebaseapp.com`.** A vendor subdomain with a
   project ID — brand leak *and* a spam signal.
3. **No SPF/DKIM/DMARC for help24.co.ke**, because you aren't sending from your domain.
   Unauthenticated mail is heavily penalised by Gmail/Outlook.
4. **No domain reputation.** help24.co.ke has no sending history.
5. **Generic templates** ("Reset your password for project-XXXX") pattern-match to phishing.
6. **`.co.ke` TLD** attracts slightly elevated filtering.
7. **No List-Unsubscribe / no engagement history.**

## PHASE 11 — Email infrastructure recommendation

| Provider | Deliverability | Free tier | Then | Verdict for Help24 |
|---|---|---|---|---|
| Firebase default | ⚠️ Poor | — | — | **Current cause of the problem** |
| **Resend** | ✅ Excellent | 3k/mo | $20/50k | ✅ **Recommended** |
| Postmark | ✅ Best-in-class | 100/mo | $15/10k | Excellent, pricier at scale |
| AWS SES | ✅ Good | 3k/mo (12mo) | **$0.10/1k** | Cheapest at scale; most setup |
| SendGrid | ⚠️ Mixed (shared IPs) | 100/day | $19.95/50k | Reputation concerns |
| Mailgun | ✅ Good | 100/day | $15/10k | Fine, unremarkable DX |
| Brevo | ⚠️ Mixed | 300/day | $25/20k | Marketing-oriented |

### → **Resend**, with **SES** as the scale-out path

**Why:** best deliverability-per-effort at Help24's stage; DNS-verified domain sending from
day one (SPF/DKIM/DMARC generated for you); 3k/mo free covers pilot; ~$20/mo at 50k;
excellent DX. **Migration path:** if you exceed ~500k emails/month, SES at $0.10/1k is
~20× cheaper — Resend is itself SES-backed, so the DNS work carries over.

**Not Postmark** only because Resend's free tier fits the pilot better; Postmark is
arguably the deliverability winner if budget is not a constraint.

### DNS configuration (Resend, domain `help24.co.ke`)

```dns
; ── SPF ── (ONE spf record only; merge if you already have one)
help24.co.ke.            TXT   "v=spf1 include:_spf.resend.com ~all"

; ── DKIM ── (exact value from Resend dashboard)
resend._domainkey.help24.co.ke.  TXT  "p=MIGfMA0GCSqGSIb3DQEB..."

; ── DMARC ── start at p=none, tighten after 2-4 weeks of clean reports
_dmarc.help24.co.ke.     TXT   "v=DMARC1; p=none; rua=mailto:dmarc@help24.co.ke; pct=100; adkim=s; aspf=s"
; after monitoring →     TXT   "v=DMARC1; p=quarantine; rua=mailto:dmarc@help24.co.ke; pct=100"
; final state →          TXT   "v=DMARC1; p=reject; rua=mailto:dmarc@help24.co.ke"

; ── Custom auth domain (removes firebaseapp.com from reset/verify links) ──
auth.help24.co.ke.       A     199.36.158.100        ; value from Firebase Hosting

; ── MX: only if you want to RECEIVE at support@help24.co.ke ──
help24.co.ke.            MX 10 mx1.improvmx.com.     ; or Google Workspace
help24.co.ke.            MX 20 mx2.improvmx.com.
```

> **Sending mail is not the same as receiving it.** SPF/DKIM/DMARC let you *send* as
> help24.co.ke. To *receive* replies at `support@help24.co.ke` you need MX records —
> use Google Workspace (~$6/user/mo) or ImprovMX (free forwarding) .
